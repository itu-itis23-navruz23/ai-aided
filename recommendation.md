In a production, high‑scale environment, this crawler–searcher architecture would need to evolve from a single‑process, thread‑based design into a distributed system. The in‑memory work queue would be replaced by a durable, horizontally scalable message broker (for example, using a task queue like Celery on top of a backend such as Redis, RabbitMQ, or Kafka) so that many crawler workers across multiple machines can coordinate URL fetching without contention. Likewise, the in‑memory `visited` set and index would be migrated to external stores: a fast key‑value database like Redis or a shared datastore for URL deduplication and crawl state, plus a search‑optimized engine such as Elasticsearch or OpenSearch for indexing titles/snippets, ranking, and querying. These changes would allow the system to handle millions of URLs, survive worker restarts, and scale out as load increases, while also enabling richer search features (fielded queries, pagination, advanced scoring, and analytics dashboards).

To operate responsibly on the open web, the crawler would also need a dedicated politeness and compliance layer. This includes parsing and respecting `robots.txt`, enforcing per‑domain rate limits and crawl delays, implementing robust retry and backoff strategies, and centralizing outbound HTTP through a configurable client (with connection pooling, timeouts, and telemetry). Around these core services, the system would benefit from an API gateway (for search and control APIs), a monitoring/observability stack (metrics, tracing, structured logs), and configuration management for crawl policies. Deployed in containers (e.g., Docker) orchestrated by Kubernetes or a similar platform, the entire pipeline—URL scheduling, crawling, parsing, indexing, and search—would be automated via CI/CD, with blue‑green or canary deployments to safely roll out changes without disrupting the production crawl or search experience.
